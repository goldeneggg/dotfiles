# OpenAI GPT-5.6 取扱説明書 (プロダクトマニュアル)

本マニュアルは、OpenAIの最新AIモデルであるGPT-5.6およびそのモデルファミリーを本番環境で安全かつ最大限に活用するための仕様、新機能、設定パラメータ、およびベストプラクティスを解説した取扱説明書です。

## 1. モデル概要とシステムラインナップ

GPT-5.6は、複雑なプロダクションワークフローにおいて、高い品質と優れた効率性を両立させる新しい基準（ベースライン）を確立したモデルです。特にトークン効率に優れており、従来よりも少ない出力トークンで最先端（フロンティア）のパフォーマンスを達成します。

用途やコスト、要求される処理速度に応じて、以下の3つのモデルラインナップから適切なモデルを選択して使用します。

- **gpt-5.6-sol (フラッグシップモデル)**: 最も高度なインテリジェンスと能力を備えたフラッグシップモデルです。一般的なAPIエイリアスである `gpt-5.6` を指定したリクエストは、自動的にこの `gpt-5.6-sol` にルーティングされます。
- **gpt-5.6-terra (高コストパフォーマンスモデル)**: 強力なパフォーマンスを維持しつつ、より低価格で利用可能な、インテリジェンスとコストのバランスに優れたモデルです。
- **gpt-5.6-luna (高効率・大量処理モデル)**: 効率性を重視したモデルであり、大量のワークロードや迅速な処理が求められるシステムに最適化されています。

## 2. GPT-5.6 の主な新機能と技術的特徴

GPT-5.6は、開発者およびエンドユーザーの体験を向上させる数多くの革新的な機能を導入しています。

### 2.1 プログラムによるツール呼び出し (Programmatic Tool Calling: PTC)

GPT-5.6は、ホストされたランタイム内でJavaScriptを記述し、指定されたツールを自動的に呼び出すことができます。この機能により、呼び出し間での結果の受け渡しや、中間出力の処理を自動で行うことができます。

- **推奨される用途**: フィルタリング、結合、ランク付け、重複排除、集約、検証など、ロジックが予測可能で、各ステップごとにモデルの新しい判断を必要としない「ツール多用型」の境界のあるワークフローに適しています。
- **メリット**: ZDR（ゼロデータリテンション）に対応しており、追加のコンテナコストは発生しません。

### 2.2 マルチエージェント機能 (Multi-agent) [ベータ版]

1つのGPT-5.6インスタンスが、並行して複数のサブエージェントを調整し、それぞれの結果を統合・合成する機能です。

独立した作業ストリームに綺麗に分割できる複雑なタスクにおいて、実時間（処理完了までの時間）を大幅に短縮し、パフォーマンスを向上させます。

現在、Responses APIにおいてベータ機能として提供されています。

### 2.3 柔軟かつ明示的なキャッシュ機能

自動のインプリシット（暗黙的）キャッシュに加え、開発者がキャッシュする再利用可能なプロンプトのプレフィックスを明示的にマークできる「明示的プロンプトキャッシュ（Explicit prompt caching）」に対応しました。

キャッシュへの書き込みは、未キャッシュの入力レートの1.25倍で課金されますが、キャッシュからの読み取りには割引が適用されます。

### 2.4 持続的な推論 (Persisted reasoning)

複数ターンにわたる会話において、前のターンで実行された推論内容（reasoning items）を再利用する機能です。

これにより、複数ターンの会話品質が向上するだけでなく、キャッシュ効率も高まります。

### 2.5 フロントエンドデザイン能力の飛躍的向上

GPT-5.6は、レイアウト、視覚的階層、デザイン判断力において強力な美的センスを備えています。これまで以上に洗練され、実用的なウェブサイトやアプリケーションのフロントエンド画面を生成することができます。

### 2.6 画像のオリジナルディテール保持 (Original image detail)

「original」または「auto」の詳細度設定で送信された画像について、従来のパッチバジェットやピクセル寸法の制限による強制的なリサイズを行わず、元の解像度・寸法を保持したまま処理します。

> **※注意**: 大きな画像は入力トークンを多く消費し、レイテンシが増加する原因になります。

## 3. 操作およびAPI設定パラメータガイド

本モデルをシステムへ統合する際、またGPT-5.4やGPT-5.5から移行する際は、以下の設定パラメータを適切にチューニングしてください。

### 3.1 接続APIの選択

推論、ツール呼び出し、および複数ターンのワークフローを構築する際は、レガシーAPIではなくResponses APIを使用してください。

### 3.2 推論努力の設定 (reasoning.effort)

GPT-5.6は、タスクの難易度に応じて以下の推論努力レベルをサポートしています。

- **none**: 推論を行わない、最速のレイテンシベースライン。
- **low**: 推論やツール使用の恩恵を受けつつ、レイテンシを最小限に抑えたいワークロード用。
- **medium**: 品質と速度のバランスが取れたデフォルトの推奨設定。
- **high / xhigh**: 推論を重ねることで明確な品質向上が得られる、難易度の高いタスク用。
- **max**: 最も困難な品質最優先のワークロード用。検証を徹底的に行います。

### 3.3 プロモード (reasoning.mode: "pro")

品質がレイテンシやトークン使用量よりも極めて重視される局面において、モデルにより多くの処理を行わせて信頼性を向上させ、単一の最終回答を出力させるモードです。

- **設定方法**: Responses APIのリクエスト内で `reasoning.mode` を `"pro"` に設定します（専用のモデルID/スラッグに切り替える必要はありません）。
- **料金**: プロモード内で消費された推論トークンは、選択したモデルの標準トークンレートで合算されて課金されます。

### 3.4 持続的推論の設定 (reasoning.context)

会話の文脈に合わせて前のターンの推論をどう扱うかを設定します。

- **auto (または未指定)**: モデルのデフォルトの挙動になります。
- **all_turns**: 会話を通じて目標や前提、優先順位が一定である場合に設定します。この設定時には `previous_response_id` を渡して以前の推論をモデルが利用できるようにします。履歴を手動管理する場合は、未暗号化の履歴に加え、`include: ["reasoning.encrypted_content"]` を追加して返された暗号化推論アイテムをリプレイします。
- **current_turn**: 過去の推論が現在のターンに関連しない、またはコンテキストをリセットしたい場合に設定します。

## 4. プロンプト設計と開発のベストプラクティス

GPT-5.6の特性を最大限に引き出すため、従来のプロンプト記述方法を見直すことを推奨します。

### 4.1 プロンプトの「スリム化（軽量化）」

GPT-5.6は、不要な重複指示や過剰な例示、複雑すぎるツールの説明を排除した、より簡潔でリーン（Leaner）なプロンプトを好みます。

**社内検証データ**: システムプロンプトをスリム化・簡素化することで、タスクの評価スコアが約10〜15%向上し、トータルトークン数が41〜66%削減、コストも33〜67%削減された実績があります。

**プロンプト最適化のステップ**:

1. 現在動作しているプロンプトとツール設定から開始します。
2. 不要な指示やツール記述をグループごとに一つずつ削除し、評価（Evals）を実行して影響を確認します。
3. 指示は一度だけ明確に記述します（繰り返しは避けます）。
4. タスクに直接関係するツールのみを公開し、その説明は簡潔かつ正確に保ちます。

### 4.2 自律性と承認境界の厳密な定義

GPT-5.6は、複数ステップのタスクを実行する際に非常に積極的かつ粘り強く取り組む傾向（プロアクティブ性）があります。

モデルが勝手に暴走したり、逆に不要な確認で処理が一時停止したりするのを防ぐため、「許可されるローカルの安全な操作（ファイルの読み取り、ログの検査、テスト実行など）」を明確なポリシーとして定義してください。

「毎回確認すること」「書き換えないこと」といった否定的な指示を何度も繰り返すと、安全な通常操作に対しても不要な承認を求めるようになるため、一箇所に整理して記述してください。

### 4.3 応答の長さ、スタイル、トーンの制御

- **簡潔さの指示の見直し**: GPT-5.6はデフォルトでGPT-5.5よりも簡潔に回答する傾向があります。プロンプト内の「簡潔に」「短く」といった指示が残っていると、回答が省略されすぎる場合があるため、移行時に必要性を再検証してください。
- **詳細度の明示的コントロール**: 一貫した長さを保つため、APIパラメータの `text.verbosity` を用いてデフォルトの詳細レベル（low / medium / high）を制御し、タスク特有の要件のみプロンプトで補足するようにします。
- **明確なトーンの定義**: 「親切に」「共感を持って」といった曖昧な表現ではなく、どのような書き方を選択するか（直接的に答えるか、問題をいつ認めるか、締めの言葉は必要かなど）を具体的に定義します。

## 5. 安全機能とセーフガード (Safeguards)

GPT-5.6を利用するにあたり、以下のリアルタイム・セーフガードの仕組みと影響について理解しておく必要があります。

### 5.1 リアルタイム悪用クラシファイア

モデル出力が生成されるのと同時に、リアルタイムのサイバー攻撃および生物学（バイオ）悪用防止クラシファイアが稼働しています。

- **動作への影響**: 悪用と判定されたリクエストはブロックまたは拒否されます。また、生成プロセスの途中で、クラシファイアによる同期レビューのために**生成が数秒間一時停止（ポーズ）**し、結果としてレスポンスに通常より時間がかかる場合があります。
- **合法的な業務への介入**: 脆弱性調査、パッチ開発、デバッグ、セキュリティ教育、防御テストといった正当な業務（デュアルユース領域）においても、初期段階の動作が攻撃的な振る舞いと酷似している場合、誤ってセーフガードが介入することがあります。

### 5.2 安全対策の実装推奨

エンドユーザー向けにアプリケーションを提供する場合は、リクエストごとに安定したプライバシー保護済みの `safety_identifier`（セーフティ識別子）を送信することで、安全性を維持しつつ誤検知を減らすことができます。
