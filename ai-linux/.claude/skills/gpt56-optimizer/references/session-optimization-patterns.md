# GPT-5.6 Codexセッション最適化パターン

## タスク別の起点

| タスク | 推奨起点 | 実行上の重点 |
| --- | --- | --- |
| 短い回答、定型変換、抽出 | Luna / Low | 入出力形式と停止条件を明確にする |
| 日常的な実装、修正、レビュー | Terra / Medium | 範囲内で実行し、対象に近い検証を行う |
| 複雑な設計、曖昧な障害、深い調査 | Sol / Medium〜High | 先に文脈を集め、判断根拠と完了条件を固める |
| 最難関の単一課題 | Sol / Max候補 | Maxが利用可能で、追加時間・使用量を許容できるか確認する |
| 分離可能な大規模作業 | SolまたはTerra / Ultra候補 | 独立性、統合点、競合しない編集範囲を定義する |
| 長時間・複数マイルストーン | Sol / `/goal`候補 | 目標、制約、検証、進捗証拠を永続化する |
| 高精度な視覚・UI作業 | Sol / Medium〜High | 既存デザインを維持し、レンダリングして確認する |

これは固定表ではない。レイテンシ、コスト、利用可能モデル、失敗時の影響、代表タスクの評価結果で調整する。

## 実行契約

次の項目のうち、挙動を変えるものだけを明確にする。

- **Outcome**: ユーザーが使える最終成果
- **Context**: 関連ファイル、資料、エラー、既存規約
- **Constraints**: 維持事項、禁止範囲、互換性、権限境界
- **Verification**: 完了を直接証明するテスト、測定、レビュー条件
- **Output**: 必須の形式、長さ、読者、証拠
- **Stop rules**: 再試行、質問、保留、終了の条件

手順そのものが要件でない限り、逐次手順を過剰に固定しない。既知の前提コマンドや壊れやすい操作順は明示する。

## 経路選択

- 目的と完了条件が明確なら直接実行する。
- 要件や設計選択が成果を大きく変えるならPlan modeで質問と計画を先に行う。
- 多数のステップ、長い待機、継続的な修復、複数マイルストーンがあるなら`/goal`を検討する。
- 一度限りの短い作業に`/goal`、小さな逐次作業にUltra、判断を伴う少数ツール呼び出しにProgrammatic Tool Callingを使わない。

## 改善例

弱い依頼:

```text
認証を直して。何をするにも確認して。
```

実行可能な契約:

```text
ログイン後にセッションが失われる原因を特定して修正し、対象テストを実行する。
認証APIの公開仕様と既存のCookie属性は維持する。ローカルの読み取り、範囲内の編集、
テスト実行は続行してよい。外部環境への書き込みや依存関係の追加前に確認する。
完了条件は、再現テストと既存の認証テストが通り、差分に無関係な変更がないこと。
```

ここで重要なのは文章の装飾ではなく、実行権限、維持事項、完了証拠が明確になったことにある。

## 完了監査

表面的な成功だけで完了としない。テスト成功が要件充足を直接証明するか確認し、必要なら差分、成果物、ログ、画面、データ状態も検査する。検証不能な項目は成功扱いせず、理由と次の確認方法を示す。
