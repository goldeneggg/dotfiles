# task-performer ルール背景（Rationale）

SKILL.md 本文から参照される、各ルールの詳細な背景説明。本文側は「ルール＋1行要約」に留め、背景や判断根拠を確認したいときにこのファイルを参照する。毎回読む必要はない。

## タスク管理コンテキストをリポジトリ成果物に残さない

**対象**: コードコメント・コミットログの両方。

**残してはいけない情報**:

- タスクドキュメントのファイルパス・URL（例: `docs/tasks/todos/001-setup/README.md`、ローカル/イントラのタスク管理URL）
- タスクID・TODO番号への言及（例: 「TODO-123対応」「タスク001の要件で追加」「YYYYMMDD-{name} プロジェクト」）
- セッション内の作業ログパス（例: `logs/NNN-{task}/...`）や、当該リポジトリ外の個人専用ディレクトリ（`.claude/agent-memory/` 等）への参照

**Why**: コードコメントとコミットログは、ソース／`git log` の一部としてリポジトリに永続化され、変更経緯を追う第三者（将来の自分・新規参画者・OSS閲覧者を含む）が読む。タスクドキュメントやタスクIDは対応者のローカル環境にしか存在せず、読み手は参照できない。残しても「どこを見ればよいか」という誤った期待を生むだけで、混乱の原因になる。コメント・コミットログは、コードとリポジトリ内の情報だけで自己完結させる。

**代わりに書くべき内容**: 「なぜ」その実装・変更にしたか（設計判断の理由・トレードオフ・非自明な前提や注意点）と、変更内容の要約（what）・リポジトリ内のファイルパスや関連機能名など、リポジトリだけを見て理解できる情報。タスク管理側の文脈は、Phase 7 の作業結果レビュー・気づきメモ・タスクファイル更新側に留める。

## テストの軽量/重い分類とローカル実行方針

**Why**: 重いテスト（DB・外部API・コンテナ起動を伴う integration/E2E）はローカルで実行環境（DB・コンテナ・認証情報・ネットワーク）の準備が必要で、環境差異による偽陽性・偽陰性が起きやすく、実行コストも高い。これらはCIの統一環境で実行する方が再現性・信頼性が高い。一方、軽量ユニットテストは環境非依存で高速なため、ローカルで即座に回す方がフィードバックループが短く有益。だからこそ「軽いものはローカル、重いものはCI」と役割を分ける。
