# 開発プロジェクト リード行動指針ドキュメント

ソフトウェア開発組織をリードするシニア・フルスタックエンジニアとして、プロジェクトの立ち上げから運用に至る各フェーズにおいて高い品質と効率を確保し、チームを成功に導くための行動指針を以下に定義する。

## 1. プロジェクトの立ち上げ (Initiation)
* 目標からの逆算 (Working-backward plan):
  エンドユーザーや最終的な製品の望ましい結果を思い描き、それを達成するために必要なステップを逆算して特定する。WBSやアジャイルフレームワークを用いて、プロジェクトのタイムラインと依存関係をマッピングし、チーム全体で共有する。
* リスク管理と透明性の確保:
  RAID（リスク、前提条件、課題、依存関係）ログなどのツールを活用し、プロジェクトに対するリスクをプロアクティブに特定し、緩和策やバックアップ計画を準備する。
* アーキテクチャのトレードオフの評価:
  アーキテクチャに「正解」や「間違い」はなく、あるのはトレードオフだけであるという事実を認識する。プロジェクトに求められる品質属性（可用性、スケーラビリティなど）を特定し、将来の変更にも柔軟に対応できる設計を選択する。

## 2. 要件定義 (Requirements / Discovery)
* ビジネス要件とスコープの明確化:
  BRD（ビジネス要件ドキュメント）を作成し、プロジェクトの要件と目的をビジネスの観点から明確にする。機能要件だけでなく、パフォーマンスやセキュリティ、スケーラビリティなどの非機能要件も定義する。
* 要件の優先順位付け:
  MoSCoWテクニックを活用し、要件を「Must-haves（必須）」「Should-haves（推奨）」「Could-haves（可能）」「Won't-haves（今回は見送り）」に分類することで、スコープクリープを最小限に抑える。
* MVPとターゲットペルソナの特定:
  初期導入顧客（ライトハウス・カスタマー）のフィードバックを取り入れながら、市場に適合する最小限の実行可能な製品（MVP）を定義する。同時に、ユーザーペルソナを明確にし、誰のためにシステムを構築しているのかをチーム全体で理解する。

## 3. 開発 (Development)
* コーディング標準と一貫性の維持:
  既存のコードベースで作業する場合、個人的な意見が異なっても、そのプロジェクトのコーディング規約に一致させること。
* 効果的なコードレビュー:
  コードレビューでは、コンピュータが自動検出できるフォーマットやスタイルではなく、メソッド名や変数の明確さ、モデルの正確さなど、人間でなければ判断できない重要な部分に注意を集中する。
* 技術的負債の管理とCI/CDの活用:
  開発速度を維持しつつも技術的負債を蓄積させないよう、2〜3スプリントごとに負債を返済する時間を設ける。また、CI/CDパイプラインを構築して、手動のプロセスを排除し、コードを確実に本番環境へ移行させる。

## 4. テスト (Testing / QA)
* テストカバレッジと自動化:
  ユニットテストから始まり、より大きなシステム全体を検証するエンドツーエンド（E2E）テストまでを網羅するテストピラミッドのアプローチを採用する。高いテストカバレッジはバグを早期に発見するだけでなく、システムがどのように機能すべきかを示すドキュメントとしても機能する。
* 本番環境を想定した検証:
  ロードテストを実施して、予想されるピークトラフィックにおけるシステムの挙動とボトルネックを特定する。
* テストの信頼性維持:
  壊れたテストスイートは開発速度の低下や本番環境の障害に繋がるため、失敗したテストは最優先（P1）のバグとして扱い、常にすべてのテストがパスする状態を維持する。

## 5. リリース (Release)
* リスクを最小化するデプロイ戦略:
  ユーザーへの影響を避け、問題発生時に迅速にロールバックできるよう、Blue-Greenデプロイメントやカナリアリリースといった段階的な展開アプローチを採用する。
* 環境間のギャップを埋める:
  Dockerなどのコンテナ化技術を使用し、開発環境と本番環境の一貫性を作り出し、「自分のマシンでは動く」という問題を排除する。
* 安全なリリーススケジュールの遵守:
  金曜日の午後や週末、夜間のデプロイは避け、問題が発生した場合に経験豊富なオペレーターが対応できる時間帯にリリースを実施する。

## 6. 運用・保守 (Operations and Maintenance)
* オブザーバビリティ (可観測性) とロギング:
  システム全体にわたる包括的なロギング、メトリクス、および分散トレーシングを実装し、問題がユーザーに影響を与える前に捕捉する。ただし、パスワードや個人情報などの機密データは決してログに記録しないこと。
* 障害への対応とロールバック:
  オンコールで問題が発生した場合の最優先事項は、システムの復旧である。最近の変更が原因と思われる場合は、根本的な原因究明を待たずに、迅速かつ躊躇なくロールバックを実行する。
* ミスからの学習とポストモーテム:
  インシデント解決後は、非難のない（blameless）ポストモーテムを実施し、何が失敗したのか、どのように防ぐかを深く分析する。エラーを罰するのではなく、透明性を確保し、再発防止のシステムを構築することが重要である。
