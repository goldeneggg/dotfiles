# GPT-5.6 Solにおける `/goal` コマンド完全攻略：自律コーディングループの最適化マニュアル

GPT-5.6 Sol（特にフラグシップモデルのSol）で提供されている `/goal` コマンドは、従来の単発プロンプトを繰り返すチャットとは異なり、開発者が指定した高いレベルの目標に向かって、エージェント自らが検証、エラー修正、自己監査を自律的に繰り返す「Ralph（ラルフ）ループ」をファーストクラスの機能として統合したものです。

この自律ループ機能を使いこなし、席を外している間（例えば 28 分間の間など）に完璧に動作する成果物を Codex にビルドさせるための具体的なコツを以下にまとめます。

---

## 1. 機能を有効化する（最初の必須ステップ）

`/goal` コマンドはデフォルトで無効化されている（または開発中ステータスである）ため、設定ファイルで明示的に有効化する必要があります 。
`~/.codex/config.toml` の `[features]` セクションに以下のフラグを設定し、Codexセッションを再起動（または立ち上げ直す）してください。

~~~toml
[features]
goals = true
~~~

※このフラグを設定しないと、TUI（ターミナルUI）で `/goal` コマンド自体が認識されません。Codex CLI 0.141.0以降であれば、ターミナルから `codex features enable goals` コマンドを実行して有効化することも可能です。

---

## 2. 仕様書駆動（Spec-Driven）でのアプローチ：3ファイルの黄金パターン

`/goal` コマンドの成否は、実行前に用意する「目標定義の精度」で9割決まります。
ただ1行の目標を投げるのではなく、リポジトリ内の `goals/<タスク名>/` ディレクトリの下に、以下の3つの役割を持った Markdown ファイルを事前に作成して配置し、Codexにこれを読み込ませる「仕様書駆動型」のパターンが最も強力です。

*   **`GOAL.md`**: 達成すべき「成果物（Outcome）」と「制約条件（Constraints）」を記述します。すべての要件に一意のID（例：`P-001`）を付与し、検証可能な完了定義（DoD）をマッピングしておきます。
*   **`VERIFY.md`**: Codexが成果物の動作を検証するために実行すべき「具体的な検証コマンド（テスト実行、linter、ビルド、型チェック等）」を定義します。
*   **`PROGRESS.md`**: 自律ループ中、Codexが要件IDごとにエビデンス（ファイルパス、テスト結果、スクリーンショットなど）を記録するための監査ログテンプレートです。

これらを事前に作成して、`/goal` コマンドにこれをターゲットにするよう指示します。

---

## 3. 「不確実なものは未完了」：プロキシシグナルを信頼させない制約

LLMエージェントが自律ループで最も起こしやすい失敗は、「テストが通過した」「書き換えた」といった表面的な記号（プロキシシグナル）だけで勝手に完了（achieved）したと勘違いし、ループを止めてしまうことです。

これを防ぐため、目標（`/goal` に渡すテキストや `GOAL.md` 内）に、Codex標準の継続用システムプロンプト（`continuation.md`）の設計思想に合致した以下の強力なフレーズを明示的に記述します。

~~~
"Treat uncertainty as incomplete. Do not accept proxy signals (like 'tests pass' or 'compiles') as completion on their own without verifying requirement fulfillment directly."
~~~

この「不確実なものは未完了として扱う」という一文を埋め込むことで、Codexは勝手に勝利宣言をせず、検証コマンドに基づく実直な監査を毎ターン行うようになります。

---

## 4. 「繰り返し回数」ではなく「トークン予算」で制限する

従来の自律ループスクリプト（自作のBashループなど）はループ回数（例：MAX_ITERS=12）で制限をかけますが、Codexの `/goal` は「トークン予算（Token Budget）」で制御されます。
タスクの複雑さに応じて、人間が意識的にトークン予算の上限を設定することが推奨されます。

*   **小規模なタスク** (単一ファイルの修正、軽微なバグ): 100k～500kトークン
*   **中規模なタスク** (複数ファイルにまたがる機能追加、テストスイート追加): 500k～2Mトークン
*   **大規模なタスク** (フレームワーク移行、大規模リファクタ): 2M～10Mトークン

予算制限に達すると、システムは自動的に `budget_limit.md` プロンプトを注入し、Codexに対して「新しい実装作業を中止し、現在の進捗を PROGRESS.md に要約して人間への引き継ぎをまとめなさい」と指示するため、予算超過時に突然プロセスが死ぬのではなく、美しく自律終了してくれます。

---

## 5. `AGENTS.md`（常時規約）と目標プロンプト（個別目標）の分離

`/goal` コマンドに渡すプロンプトに、フレームワークのルールやリポジトリ固有のビルド手順を詰め込みすぎると、毎ターンの文脈が肥大化し「コンテキストの腐敗（context rot）」やトークンの無駄遣いが発生します。

*   **`AGENTS.md` (会社ハンドブック/プロジェクトの不変ルール)**: テクニカルスタック、テスト実行コマンド、コード規約など、すべてのターンで常時参照すべき固定ルールを記述しておきます。
*   **`/goal <目標>` (個別のジョブチケット)**: その特定の自律実行タスクでのみ考慮すべき「対象範囲、振る舞いの契約、非目標、検証パス」に特化させ、4,000文字の上限に収まるように記述します。

このように階層化することで、エージェントは迷わずに最も効率的な推論パスを見つけ出します。

---

## 6. 自律を加速する「フルオートモード」の設定

自律実行中にCodexがコマンドを生成するたびに確認（プロンプト承認）を求められて一時停止すると、放置自律が活かせません。
そのため、基本的には `full-auto` 承認モード（または承認ポリシーを `on-request` にし、自動承認エージェントである `auto_review` を設定する）で使用するのがコツです。

~~~toml
# ~/.codex/config.toml
approval_policy = "on-request"
approvals_reviewer = "auto_review"
~~~

これにより、サンドボックス内の安全なローカルコマンド（テストの実行やファイル読み込みなど）は自動実行させ、危険な破壊的操作や特定の外部接続時のみに人間の承認を求める、高効率かつ安全なパイプラインが構築できます。

---

## 7. コンテキスト圧縮による状態消失（Issue #19910）の回避策

タスクが長期化すると、コンテキスト制限を維持するためにシステムが過去履歴を圧縮（Compaction）します。
古いバージョン（0.128.0前後）では、コンテキスト圧縮時に実行中のゴール目標や監査ルール（`continuation.md`）の文脈が消失してしまう致命的なバグが報告されていました。

*   **対策1**: CLIクライアントを最新（0.138.0以降）にアップデートしてください。コンテキスト圧縮におけるゴールのハンドリングが大幅に改善されています。
*   **対策2**: 大規模な10Mトークンのゴールを設定するのではなく、1M〜2Mトークン規模の小さな段階的なマイルストーンに分割し、複数の目標を順に起動してください。
*   **対策3**: もし状態がスタックしたり混乱した場合は、一度 `/goal clear` で状態をクリアし、再び `/goal <元の目標>` を実行することで、ワークスペース上のコードを維持したまま、クリーンな状態でループを再開できます。

## シニア・ソフトウェア開発者としての推論と総括

以上を踏まえると、GPT-5.6 Sol環境における `/goal` コマンドは、単なる「無限ループするシェルスクリプト」ではなく、「**検証可能なDoD契約（ゴール仕様）の自動監査エンジン**」として機能させることが本質であると考えられます。

人間がコードの実装ステップを1から10まで指示する（マイクロマネジメントする）のをやめ、テスト、仕様ファイル等の「検証基準」をCodexに噛み合わせるための『優れた仕様策定者（Spec Writer）』に回ることで、開発効率（Vibe CodingのROI）は劇的に最大化されると結論づけられます。
