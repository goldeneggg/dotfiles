# 優れたAgent Skills SKILL.mdの共通ポイント

優れたAgent SkillsのSKILL.mdに共通しているポイントは、AIエージェントが**「いつそのスキルを使うべきか」を正確に判断でき、コンテキストウィンドウ（記憶容量）を無駄に消費せず、タスクを確実かつ自律的に遂行できるように設計されている**ことです。

優れたSKILL.mdの共通ポイントを6つの重要な要素に分けて詳しく解説します。

## 1. 正確なトリガーを促すメタデータ（Description）の最適化

スキルがどれほど優れていても、エージェントが適切なタイミングで呼び出さなければ意味がありません。優れたスキルは、フロントマター（YAML）のdescriptionフィールドを「スキルの要約」としてではなく、「モデルに対するトリガー条件」として記述しています。

- **三人称で記述する**: "Use this skill when..."（このスキルを使って…）ではなく、"This skill should be used when..."（このスキルは…の時に使用されるべきです）といった客観的な三人称視点で記述します。
- **具体的なトリガーフレーズを含める**: ユーザーが実際に発話する可能性のある具体的なキーワードやシナリオ、症状などを記載します。たとえば「Excelファイルを処理します」よりも、「ユーザーが『売上CSVを分析してチャートを作成して』など、CSV、TSV、Excelデータの集計や派生カラムの作成を求めた時に使用します」のように具体的に書きます。
- **過度な要約を避ける**: スキルの内部ワークフローや手順そのものをdescriptionに要約してはいけません。モデルが本文を読まずに、要約だけを頼りに作業を進めてしまう（ショートカットしてしまう）リスクがあるためです。

## 2. 段階的開示（Progressive Disclosure）によるコンテキスト管理

エージェントのコンテキストウィンドウは貴重なリソースです。優れたスキルは「最初からすべてを読み込ませる」のではなく、**必要な情報を必要な時にだけ読み込ませる構造（Progressive Disclosure）**を採用しています。

- **SKILL.md本体はスリムに保つ**: コアとなるSKILL.mdは500行以内（理想的には1500〜2000語程度）に収め、必要最小限の指示のみを記述します。
- **詳細情報は別ファイルに分割する**: 詳細なAPIリファレンス、データベースのスキーマ、長大なテンプレート、実行スクリプトなどは、`references/`、`examples/`、`scripts/`、`assets/` といったサブディレクトリに分割します。
- **「いつ読むべきか」を明記する**: 単に「リファレンスを参照」と書くのではなく、「APIが200以外のステータスコードを返した場合は、`references/api-errors.md` を読むこと」のように、エージェントが特定のファイルを取得すべき条件をSKILL.md内に明確に指示します。

## 3. 客観的で明確な指示の記述スタイル

エージェントに対する指示（プロンプト）の書き方にも、優れたスキルならではの共通したトーンとスタイルがあります。

- **命令形 / 不定詞を使用する**: 「あなたは〇〇すべきです（You should do X）」といった第二人称ではなく、「〇〇を達成するには、××を実行する（To accomplish X, do Y）」のような、動詞から始まる客観的で指示的な表現を使用します。
- **「なぜ（Why）」を説明する**: 厳格な「必ず〇〇しろ（ALWAYS do X）」という命令よりも、「〇〇を行うこと。なぜなら××という問題を引き起こすため」と理由を説明する方が、AIモデルはその意図を理解し、コンテキストに応じた適切な判断を下しやすくなります。
- **プロジェクト固有の「Gotchas（落とし穴）」を記載する**: 「エラーを適切に処理する」といった一般的なベストプラクティスではなく、そのプロジェクトやAPI特有の非自明なエッジケースや、モデルが間違いやすいポイント（Gotchas）を具体的に列挙します。

## 4. 具体性と柔軟性のバランス（適切な自由度の設定）

タスクの性質（壊れやすさや許容されるバリエーション）に応じて、指示の「自由度（Degrees of freedom）」を適切に調整しています。

- **高い自由度（テキストベースの指示）**: 複数のアプローチが有効で、コンテキストに依存するタスク（コードレビューなど）では、考え方の指針だけを与えてエージェントの推論能力に委ねます。
- **低い自由度（スクリプトや厳格な制約）**: 順序が絶対に重要であるタスクや、少しのミスでシステムが壊れるような脆弱な操作（データベースのマイグレーションなど）では、エージェントに考えさせるのではなく、確定的（Deterministic）なユーティリティスクリプトを呼び出させるか、厳格な手順を強制します。

## 5. ワークフローと自己検証ループの組み込み

複雑なタスクを確実にこなすため、優れたスキルは作業を細分化し、エージェント自身に確認させるプロセスを持っています。

- **チェックリストの提供**: 複数のステップや依存関係があるワークフローでは、明確なチェックリストを提供し、エージェントがステップを飛ばさないようにします。
- **自己検証ループ（Validation loops）**: 「作業を行う → バリデーション（スクリプトやリファレンスとの照合）を実行する → 問題を修正する → パスするまで繰り返す」というパターンを指示に組み込み、出力の品質を担保します。
- **Plan-Validate-Execute（計画・検証・実行）**: 一括処理や破壊的な変更を行う場合は、まず構造化された中間プラン（計画）を作成させ、それを真の情報源（Source of Truth）と照らし合わせて検証してから、初めて実行に移させるように指示します。

## 6. テスト駆動開発（TDD）のアプローチによるスキルの改善

最も優れたスキルは、想像だけで書かれるのではなく、実際のAIの振る舞いを観察し、テストと反復（Evaluation-driven development）を経て構築されています。

- **失敗の観察（RED）**: まずスキルがない状態でエージェントにタスクをやらせてみて、どのような言い訳（Rationalization）をしてルールを破るか、どう失敗するかを正確に記録します。
- **最小限のスキルの作成（GREEN）**: 観察された具体的な失敗を防ぐためだけの、必要最小限の指示をSKILL.mdに書きます。起こりもしない仮定のシナリオに対する余分な指示は書きません。
- **抜け穴の封鎖（REFACTOR）**: スキルを与えた状態でエージェントにプレッシャー（時間的制約や疲労など）をかけ、それでもルールを破る新しい言い訳を見つけた場合は、それを防ぐ明確なカウンター（対抗策）をスキルに追記し、防弾（Bulletproof）化します。

## まとめ

総じて、優れたSKILL.mdは、**「AIエージェントにとって最も読みやすく、理解しやすく、そして確実に実行できるマニュアル」**として、ソフトウェアエンジニアリングの原則（関心の分離、テスト駆動、再利用性）をプロンプトエンジニアリングに応用したものだと言えます。
