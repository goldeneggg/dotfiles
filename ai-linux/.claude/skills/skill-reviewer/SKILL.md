# Skill Reviewer

既存スキルの品質をレビューし、チェックリストに基づいて改善提案を行う。

## ゴール

開発されたスキルを本番環境で使用可能なレベルに引き上げる。

## Degrees of Freedom

- **チェックリスト評価: Low freedom** — チェック項目は `references/skill-check-list.md` で定義済み。項目を飛ばさず、判定基準に従って ✅/⚠️/❌ を機械的に付与する
- **改善提案の内容: High freedom** — 対象スキルの目的・構造・コンテキストに応じて、最適な改善案を判断する。チェックリストにない問題も指摘してよい

## ユースケース

```
ユースケース: スキル品質レビュー
トリガー: 「このスキルをレビューして」「/skill-reviewer [name]」
          「公開前にチェックして」「改善点を提案して」
手順:
1. 対象スキルのディレクトリを特定し、全ファイルを読み込む
2. チェックリストに基づき4フェーズで評価
3. フェーズごとに結果を対話的に提示
4. 優先度付きアクションリストで総括
成功基準: 具体的な改善アクションが優先度付きで提示され、
          ユーザーが追加質問なしで改善に着手できる状態になること
```

## ワークフロー

1. **スキル情報の取得**: ユーザーにレビュー対象スキルを確認
2. **ファイル読み込み**: SKILL.md、references/、scripts/、assets/を並列で探索・読み込み
3. **チェックリスト評価**: 各フェーズの項目を順次確認
4. **対話形式レポート**: 問題点と改善提案を段階的に提示
5. **優先度付け**: Critical/High/Medium/Lowで優先順位を提示

## チェックリスト

評価前に `references/skill-check-list.md` を読み込む。内容:

- **Sub Agent vs Skill の妥当性評価** — 隔離・権限分離・並列処理の必要性を判定
- **事前準備フェーズ** — 要件理解、ユースケース定義、リソース計画、メタデータ設計、実装スタイル、サブエージェント方式、トリガー検証
- **実装フェーズ** — frontmatter完全性、body明瞭性（500行以内）、resources構成
- **検証フェーズ** — トリガー動作、エラーハンドリング、出力形式
- **公開前フェーズ** — ドキュメント完全性、保守性、トリガーテストスイート

Sub Agent判定が必要な場合は `references/subagent-vs-skill.md` の「使い分けのポイント」「補足: クイックチャート」を参照。

## レビュー実行プロセス

### ステップ1: スキル特定とファイル探索

**レビュー対象**: $ARGUMENTS

引数が未指定の場合はAskUserQuestionツールでユーザーに確認。

1. スキルのディレクトリパスを特定（例: `.claude/skills/$ARGUMENTS/`）
2. Globで以下を**並列探索**: `SKILL.md`、`references/**`、`scripts/**`、`assets/**`
3. 発見したファイルを**並列Read**で読み込み（SKILL.md + 各referencesファイルを同時に）

### ステップ2: チェックリスト評価

進捗チェックリストをコピーして追跡する:

```
Review Progress:
- [ ] Phase 1: 事前準備フェーズ評価
- [ ] Phase 2: 実装フェーズ評価
- [ ] Phase 3: 検証フェーズ評価
- [ ] Phase 4: 公開前フェーズ評価
- [ ] Summary: 総合評価・優先アクション提示
```

各項目の判定:
- ✅ **PASS**: 基準を満たしている
- ⚠️ **WARNING**: 改善推奨
- ❌ **FAIL**: 改善必須

評価ルール:
- フェーズごとに区切って段階的に評価（一度に全フェーズを出力しない）
- 「不足」ではなく「SKILL.mdのX行目にY情報がない」と具体的に指摘
- 問題点には具体的な改善例を提示

### ステップ3: 対話形式レポート提示

フェーズごとに結果をセクション分けして提示。問題検出時は即座に改善提案を提示し、「この部分を改善しますか?」と確認する。

出力ルール:
- 絵文字で視認性向上: 📋 🔍 📝 🧪 📤 📊 🎯 ✅ ⚠️ ❌ 🔴 🟠 🟡 🟢
- フェーズごとに `---` で区切る
- 改善案はコードブロックや箇条書きで具体的に提示
- 一度に10個以上の問題を提示しない（重要度順に段階的提示）

```markdown
## 📋 レビュー開始: [スキル名]

**概要**: [スキルの目的を1-2文で要約]

---

## 🔍 1. 事前準備フェーズの評価
[各項目について ✅/⚠️/❌ で判定。問題には具体的な改善案を提示]

---
## 📝 2. 実装フェーズの評価
---
## 🧪 3. 検証フェーズの評価
---
## 📤 4. 公開前フェーズの評価

---

## 📊 総合評価

### 問題サマリー
- 🔴 Critical: X件
- 🟠 High: X件
- 🟡 Medium: X件
- 🟢 Low: X件

### 🎯 優先改善アクション
[優先度順に具体的なアクション（実施内容と期待効果を明記）]
```

## エラーハンドリング

### Target skill not found
指定されたスキルディレクトリが存在しない場合:
1. `.claude/skills/` 配下で類似名を Glob で検索
2. 候補をユーザーに提示して確認

### SKILL.md has invalid frontmatter
YAML解析に失敗した場合:
1. 具体的なYAMLエラー箇所を報告
2. レビューの最初のFAIL項目として記録し、残りの評価を続行

### Arguments not provided
引数未指定で実行された場合:
1. AskUserQuestionツールでレビュー対象スキルを確認

### Referenced files not readable
references/内のファイルが読めない、または存在しない場合:
1. 読めたファイルのみでレビューを続行
2. 欠損ファイルをWARNINGとして報告

## 注意事項

### トークン効率

- チェックリストは `references/skill-check-list.md` をReadツールで読み込み、最新の内容を取得
- 対象スキルのreferences/が複数ある場合、内容を推測してから選択的読み込み
- SKILL.mdと対象スキルのファイル群はGlobで探索→必要箇所のみRead

### Progressive Disclosure原則

- 全項目を一度に評価せず、フェーズごとに区切る
- 問題が多い場合は重要度順に段階的提示（一度に10個以上の問題を提示しない）
- 詳細なチェック基準は`references/skill-check-list.md`に委譲

### 具体性

- 抽象的指摘（「不十分」「改善が必要」）ではなく、具体的な問題箇所と改善案を提示
- 改善案はコード例や文言例で示す
- 優先度の根拠を明確に説明（「なぜHighなのか」）

### 対話形式の重視

- 一方的なレポート出力ではなく、ユーザーとの対話を促す
- 問題検出時は「この部分を改善しますか?」と確認
- 優先度の高い問題から順に提示し、ユーザーの反応を見て次に進む
