# Task Starter プロンプトテンプレート

`task-starter` の入力を事前に整理するときは、以下をコピーして分かる範囲で埋める。未決定の項目は `未定` と記載してよい。空欄や `未定` があってもスキルを開始し、不足情報はエージェントが段階的に確認する。

## 情報収集ルール

1. ユーザーの依頼、`$ARGUMENTS`、対象コードから既知の情報を抽出する。
2. 既存コードから確認できる項目は、質問する前に対象ファイルを調査する。
3. 下記の「生成開始条件」と照合し、不足情報を重要度順に1回1〜3問ずつ質問する。
4. 選択可能な質問には推奨案と判断材料を添える。回答済みの項目は再質問しない。
5. 任意項目が未定の場合は合理的な案を提示し、成果物が大きく変わる場合だけ確認する。
6. 収集結果と採用する前提を要約し、ユーザーの確認後にファイル生成へ進む。

## 生成開始条件

以下が確定するまで Phase 2 のファイル生成へ進まない。

- プロジェクト/タスク名
- 出力先ディレクトリ
- 概要、背景・目的
- 新規開発または既存コード改修の区分
- 対象範囲と、タスク分割に必要な主要要件
- 会話上で検証可能な完了条件または検証方法
- ドキュメント形式（Markdown / HTML）
- 参考データ・ファイルの有無
- 既存コード改修の場合は、対象リポジトリ/ディレクトリと現状の課題

技術スタック、UI/API、データモデル、外部依存、非機能要件は、対象コードや要件から判断できる場合はエージェントが案を作る。重要な選択肢が複数ある場合だけユーザーへ確認する。

## 不足情報の質問順

一度に全項目を質問せず、回答によって不要になる質問を省く。

1. 基本情報: 名前、目的、種別
2. 対象: 主要要件、対象範囲、既存コードの場所
3. 完了条件: 期待動作、検証方法、重要な制約
4. 出力: 出力先、形式、参考ファイル

ユーザーが完了条件を具体化できない場合は、対象コードのテスト・lint・ビルドコマンドや期待動作を基に検証可能な候補を提示し、合意を取る。
