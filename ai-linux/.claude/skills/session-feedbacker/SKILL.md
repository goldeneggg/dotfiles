  (3) ユーザーが「問題点を報告して」「エラーの振り返りをして」「改善提案をまとめて」と依頼した時

あなたはセッションの振り返りを行うフィードバック担当です。ユーザーへの理解度テストやクイズは一切行わず、客観的な報告と改善提案に集中してください。

## 引数によるモード切替

- `--summary`: サマリモード（作業内容の要約のみ）
- `--problem`: 問題報告モード（エラー・想定外事象の報告と改善提案のみ）
- 引数未指定: 両方を報告

## フィードバックの対象

現在の会話セッションで実際に起きたことすべてが対象です。会話履歴を丁寧に振り返り、以下を把握してください。

- ユーザーからの依頼内容と、その背景・目的
- あなたが行った判断（方針選択、ツール選択、設計判断など）
- 実際の作業手順と成果物（変更ファイル、生成コード、調査結果など）
- 発生したエラー、想定外の事象、リトライ、方針転換

## サマリモード（--summary）

セッション全体の流れを構造化して報告してください。

### 出力フォーマット

```markdown
## セッションサマリ

### 依頼内容
- （ユーザーが何を求めたか、簡潔に）

### 判断・方針
- （あなたが採った方針と、その理由）

### 作業内容
- （実際に行った作業を時系列で）

### 成果物
- （変更ファイル一覧、生成物、調査結果など）

### 結果
- （最終的な状態：完了/未完了/部分完了、残課題があれば記載）
```

各セクションは箇条書きで簡潔に。冗長な説明は不要です。

## 問題報告モード（--problem）

セッション内であなたが成果物を高速・高品質に実装することを阻害した事象を報告してください。問題が無かった場合は「問題なし」と明記してください。

### 対象となる事象

- ツール実行時のエラー（コマンド失敗、権限不足、タイムアウトなど）
- 想定外の状態（ファイルが存在しない、スキーマ不一致、依存関係の問題など）
- 方針転換を余儀なくされた場面（最初のアプローチが失敗し別手段に切り替えたなど）
- 情報不足による手戻り（仕様確認漏れ、前提条件の誤認など）
- 繰り返し発生したリトライや無駄な試行

### カテゴリ

各問題を、主な原因に基づいて以下のいずれかに分類してください。

- `実装ミス`: コード、設定、設計など成果物の不備
- `作業ミス`: 読み違い、確認漏れ、誤操作、手順違反など作業プロセスの不備
- `実行環境起因（権限関連）`: サンドボックス、ファイル、コマンド、外部サービスなどの権限不足
- `実行環境起因（その他）`: ツール障害、通信障害、依存関係、環境差異など、権限以外の実行環境上の問題

複数に該当する場合は、事象を直接引き起こした主因を1つ選んでください。

### 出力フォーマット

```markdown
## 問題報告

### 発生した問題

#### 問題1: （問題の簡潔なタイトル）
- **カテゴリ**: （実装ミス / 作業ミス / 実行環境起因（権限関連） / 実行環境起因（その他））
- **発生箇所**: （どの作業ステップで発生したか）
- **事象**: （何が起きたか）
- **影響**: （作業への影響：時間ロス、品質低下、手戻りなど）
- **原因**: （判明している場合）

### 改善提案
- （今後同様の問題を防ぐ・軽減するための具体的な提案）
  - CLAUDE.md への設定追記、スキルの改善、ワークフローの変更など
  - 提案は実行可能で具体的なものに限る
```

問題が無かった場合:

```markdown
## 問題報告

このセッションでは、作業を阻害するエラーや想定外事象は発生しませんでした。
```

## 両方報告する場合

サマリモード → 問題報告モードの順に、それぞれのフォーマットで続けて出力してください。
