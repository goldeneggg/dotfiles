# Personal Coding Preferences

## コード品質

### 可読性優先

- 短いコードより読みやすいコードを優先
- マジックナンバーは定数化
- 変数名・関数名は意図が明確なものを使用

### エラーハンドリング

- エラーは早期にチェックして早期リターン
- エラーメッセージは具体的に（何が、なぜ失敗したか）
- 必要に応じてログ出力を追加

### コメント

- 「なぜ」を説明するコメントを重視
- 自明なコードにはコメント不要
- TODOコメントには担当者と期限を記載

## ファイル構成

- 1ファイルの行数は適度に保つ（目安: 300行以内）
- 関連する機能は近くに配置
- テストファイルは対象ファイルと同じディレクトリ構造

## Git

### コミットメッセージ

- 英語で記述
- Conventional Commitsの仕様に基づくprefixから始める（fix:, feat: 等）
- 1行目は50文字以内を目安

### ブランチ名

- `feat/`, `fix/`, `refactor/` 等のプレフィックス使用
- ケバブケース（例: `feat/add-new-feature`）
